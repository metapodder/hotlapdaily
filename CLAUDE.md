## Architecture

**Hotlap Daily** is a Next.js 15 / React 19 daily racing game. Players set lap times on procedurally-generated F1 tracks; times are submitted to a global leaderboard.

### Two distinct code layers

**1. React/Next.js app** (`/src/`): Server-rendered shell, API routes, admin dashboard. Key files:
- `src/app/page.tsx` — Main game page (~1180 lines); renders canvas and initializes the game
- `src/app/api/best-lap/route.ts` — Lap submission with anti-cheat, PoW, HMAC validation
- `src/lib/trackGenerator.ts` — Procedural track generation
- `src/lib/trackValidator.ts` — Server-side track/lap validation
- `prisma/schema.prisma` — 5 models: BestLap, SubmittedTrack, TrackFunction, Wrapped, Feedback

**2. Client-side game engine** (`/public/game/`): Vanilla JS modules served as static assets, organized by concern. The game runs independently of React once initialized. Imports use absolute paths (e.g. `/game/render/draw.js`).
- `public/game/engine.js` — `Game` orchestrator: core loop, lap tracking, crash detection, rendering + ghost clusters (~2400 lines)
- `public/game/physics/controller.js` — `CarController`: time-based physics, keyboard/touch input
- `public/game/physics/physics.js` — physics-constant capture/validation
- `public/game/render/draw.js` — Canvas rendering (pixel-art car, track centerline, arrows); `render/colors.js` — color helpers
- `public/game/anticheat/anticheat.js` — `AntiCheatSystem`: checkpoint validation, line-crossing detection
- `public/game/net/api.js` — All `/api/*` fetch calls; `net/telemetry.js` — telemetry
- `public/game/tracks/tracks.js` — track generator functions + client-side fallback track
- `public/game/ui/{ui,leaderboard,share}.js` — HUD, in-game leaderboard, share-card image
- `public/game/util/geometry.js` — pure geometry helpers

The entry module is loaded by `page.tsx` as `<script type="module" src="/game/engine.js">`. See `ARCHITECTURE.md` for the full layer map.

### Data flow

```
page.tsx renders <canvas> → /game/engine.js initializes → requestAnimationFrame game loop
→ user completes lap → anticheat validates → net/api.js POSTs /api/best-lap
→ route.ts validates (PoW + HMAC + physics) → pluggable data layer (memory | PostgreSQL)
```

### Pluggable persistence

All routes import a single `prisma` client from `src/lib/prisma.ts`, which
selects a backend at runtime: an in-memory store (`src/lib/db/memoryClient.ts`,
the zero-setup default) or real Prisma/PostgreSQL when `DATABASE_URL` is set.
Force one with `DB_BACKEND=memory|postgres`. Route code is backend-agnostic.

### Anti-cheat

Lap submissions require: (1) Proof-of-Work challenge/response from `/api/challenge`, (2) HMAC-SHA256 signature of physics constants, (3) server-side checkpoint coverage check (≥50% of checkpoints visited in order). The `AntiCheatSystem` class in `anticheat.js` tracks this client-side; the server re-validates.

### Track generation

Tracks are procedurally generated by functions stored as code strings in the `TrackFunction` DB table. The server fetches a function from the DB and sends it to the client; the client executes it via `new Function('return ' + trackCode)()` to produce a path array. A new track is selected daily.

### State persistence

Client state lives in localStorage (driver name, constructor choice, ghost enabled, test mode flag). Ghost replay data is a sampled trail of `{x, y}` coordinates at 50ms intervals, stored in DB via `/api/ghost-trace`.

### Environment variables

All optional — the app runs with no `.env` using the in-memory backend.
- `DATABASE_URL` (PostgreSQL) — opt into Postgres
- `DB_BACKEND` (`memory`|`postgres`), `DB_PERSIST` (`false` to disable `.data/`)
- `PASSWORD_ADMIN` — admin dashboard password
- `HOTLAPDAILY_CHALLENGE_SECRET` — stable anti-cheat signing secret
See `env.example`.
